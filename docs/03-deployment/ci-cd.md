# CI/CD Pipeline

## Workflows

- `ci.yml`
Runs on pull requests and branch pushes (`staging`, `main`) to validate extension code and docs.

- `staging-version-guard.yml`
Runs on pull requests to `staging` and blocks release-version changes (`extension/manifest.json` and docs index version line).  
Version bumping is release-only and must happen from `main` automation.

- `deploy-docs.yml`
Runs on `staging` to build and deploy MkDocs docs to GitHub Pages.

- `release.yml`
Runs on `main` to:
1. auto-bump semantic version (default `patch`)
2. update release version files (`extension/manifest.json`, docs index version line)
3. create release commit (`chore(release): cut vX.Y.Z`)
4. create tag `vX.Y.Z`
5. generate GitHub release notes automatically
6. package extension zip
7. upload + publish to Chrome Web Store API

- `weekly-staging-to-main.yml`
Runs weekly (and on manual dispatch) to create/reuse a sync PR from `staging` to `main` when `staging` is ahead.
It requests auto-merge and can optionally use admin fallback merge when `REPO_ADMIN_TOKEN` is configured.

- `apply-branch-protection.yml`
Manual workflow to apply branch protection policies for `main` and `staging` using admin token (`REPO_ADMIN_TOKEN`).

## CI Validation Gates

`ci.yml` enforces:

1. `node --check` on all extension JavaScript files
2. JSON validation for `extension/manifest.json`
3. extension package integrity validation (`node scripts/validate-extension.mjs`)
4. junk-file rejection (`.DS_Store`)
5. strict MkDocs build (`mkdocs build --strict`)

## Chrome Web Store Publish Flow

1. Merge cycle promotes code from `staging` to `main`.
2. Release workflow computes next semantic version automatically.
3. Workflow commits version bump to `main`.
4. Workflow creates and pushes tag `v<version>`.
5. Workflow publishes GitHub Release with autogenerated release notes.
6. Workflow packages extension zip in `dist/`.
7. Workflow uploads extension package to Chrome Web Store API.
8. Workflow publishes uploaded package (`default` target unless overridden by `CWS_PUBLISH_TARGET`).

## Required Secrets

- `CWS_CLIENT_ID`
- `CWS_CLIENT_SECRET`
- `CWS_REFRESH_TOKEN`
- `CWS_PUBLISHER_ID`
- `CWS_EXTENSION_ID`
- optional `CWS_PUBLISH_TARGET`
- optional `REPO_ADMIN_TOKEN` (for branch-protection apply + weekly admin merge fallback)
